1. Resum executiu

La solució desplega AutoCo en un clúster de tres nodes Proxmox VE amb contenidors LXC Ubuntu Server 24.04. L'arquitectura s'organitza en tres capes independents, cadascuna dissenyada per al mínim consum de recursos i la màxima tolerància a fallades del hardware.

🟢 Capa 1 · Publicació

Dos nodes actiu/passiu amb IP flotant (Keepalived VRRP). Traefik gestiona el balanceig de càrrega amb sticky sessions per a Blazor Server. Failover en <5 s.

🟠 Capa 2 · Dades

Clúster PostgreSQL de 3 nodes gestionat per Patroni + etcd. Failover automàtic en <30 s. Redis amb Sentinel. pgBouncer com a proxy transparent.

🟣 Capa 3 · Monitorització

Prometheus + Grafana + Alertmanager al node pve-3. Alertes per correu electrònic. Harbor Registry aïllat al mateix node físic.

Restricció de xarxa: tota la solució utilitza 6 IPs de la xarxa 192.168.0.0/24 (3 nodes Proxmox + 2 nodes de publicació + 1 VIP de servei). Tota la comunicació interna entre capes usa la xarxa 10.10.0.0/24, invisible des de la LAN escolar.

6. Disseny de xarxes Proxmox

Bridges Linux (fase inicial — un sol node físic)

En la fase inicial, tots els LXC corren al mateix Proxmox. Els bridges són locals:

BridgeTipusRangPropòsit
vmbr0Bridge extern192.168.0.0/24Uplink al switch físic → internet
vmbr1Bridge intern10.10.0.0/24Comunicació inter-contenidors (sense uplink)

Configuració /etc/network/interfaces (per node)

auto lo
iface lo inet loopback

# NIC física
auto enp3s0
iface enp3s0 inet manual

# Bridge extern (LAN escolar)
auto vmbr0
iface vmbr0 inet static
    address 192.168.0.10/24     # .11 per pve-2, .12 per pve-3
    gateway 192.168.0.1
    bridge-ports enp3s0
    bridge-stp off
    bridge-fd 0

# Bridge intern (inter-contenidors, sense uplink)
auto vmbr1
iface vmbr1 inet static
    address 10.10.0.254/24      # IP del node al bridge intern
    bridge-ports none
    bridge-stp off
    bridge-fd 0

Fase producció — 3 nodes separats

Quan es distribueix en 3 nodes físics, vmbr1 (xarxa interna) ha d'estendre's entre nodes. Opcions per ordre de preferència:

OpcióRequisitCom
VLANs 802.1QSwitch gestionatTrunk VLAN al switch, vmbr0 VLAN-aware, tag VLAN 10 a vmbr1
2a NIC dedicada2a targeta de xarxavmbr1 amb uplink a la 2a NIC en xarxa isolada
WireGuard overlayCap (software)Túnel xifrat entre nodes sobre vmbr0. Solució temporal fins tenir switch gestionat.

7. Pla d'acció fase a fase

Cada fase és independent i verificable. Es pot aturar entre fases sense deixar el sistema en un estat inconsistent.

Preparació del clúster Proxmox

Prerequisit de tot · Sense downtime

Creació dels contenidors LXC base

Tots els 7 contenidors · Ubuntu 24.04 · Docker instal·lat
Temps estimat: 1-2 h Downtime: cap Prerequisit: Fase 0 completada

Crear cada LXC amb la comanda següent (ajustar VMID, hostname, IPs i node destí):

# Exemple per a ct-autoco-pub-01 (VMID 101, a pve-1)
pct create 101 local:vztmpl/ubuntu-24.04-standard_*.tar.zst \
  --hostname ct-autoco-pub-01 \
  --cores 2 --memory 2048 --swap 512 \
  --rootfs local-lvm:10 \
  --net0 name=eth0,bridge=vmbr0,ip=192.168.0.101/24,gw=192.168.0.1 \
  --net1 name=eth1,bridge=vmbr1,ip=10.10.0.11/24 \
  --features nesting=1,fuse=1 \
  --unprivileged 0 \
  --start 1
  1. Crear els 7 contenidors seguint la taula d'especificació (secció 5)
  2. Instal·lar Docker Engine a cada contenidor: curl -fsSL https://get.docker.com | sh
  3. Instal·lar docker compose, curl, htop, vim a cada contenidor
  4. Configurar SSH key-based auth des de la màquina de gestió
  5. Verificar connectivitat interna: ping 10.10.0.21 des de pub-01
Docker en LXC privilegiat: cal nesting=1 i unprivileged=0 per a Docker. Afegir a /etc/pve/lxc/<vmid>.conf: lxc.apparmor.profile: unconfined

Capa de dades — PostgreSQL HA + Redis

Patroni + etcd (3 nodes) · Failover automàtic · Redis Sentinel
Temps estimat: 3-4 h Downtime: cap (nova infraestructura) Prerequisit: Fase 1 completada
  1. Desplegar clúster etcd als 3 nodes db via Docker Compose (ports interns 2379/2380)
  2. Verificar quòrum etcd: etcdctl endpoint health --endpoints=10.10.0.21:2379,10.10.0.22:2379,10.10.0.23:2379
  3. Desplegar Patroni + PostgreSQL 16 als 3 nodes db. Patroni gestiona l'elecció del primari via etcd.
  4. Verificar clúster Patroni: patronictl -c /etc/patroni.yml list → 1 Leader + 2 Replica
  5. Desplegar Redis master (db-01) + replica (db-02) + Sentinel (db-01, db-02)
  6. Desplegar pgBouncer a db-01 i db-02, apuntant al primari PostgreSQL actual
  7. Configurar Keepalived als nodes db per al Data VIP 10.10.0.100 → segueix el primari Patroni

Estructura docker-compose.db.yml (esquema)

services:
  etcd:
    image: quay.io/coreos/etcd:v3.5
    # ...configuració cluster etcd

  patroni:
    image: patroni/patroni:latest
    environment:
      PATRONI_NAME: db-01
      PATRONI_POSTGRESQL_DATA_DIR: /data/patroni
      PATRONI_ETCD3_HOSTS: "10.10.0.21:2379,10.10.0.22:2379,10.10.0.23:2379"
      PATRONI_POSTGRESQL_CONNECT_ADDRESS: "10.10.0.21:5432"
    volumes:
      - pg-data:/data/patroni
      - ./patroni.yml:/etc/patroni.yml

  pgbouncer:
    image: pgbouncer/pgbouncer:latest
    # apunta sempre al Data VIP 10.10.0.100

  redis:
    image: redis:7-alpine
    # master a db-01, replica a db-02
Verificació final: matar el contenidor patroni a db-01 → Patroni ha de promoure db-02 a primari en <30 s sense intervenció manual.

Capa de publicació — nginx + Traefik + Keepalived

VIP actiu/passiu · Sticky sessions Blazor · SSL
Temps estimat: 2-3 h Downtime: cap (nova infraestructura) Prerequisit: Fase 2 completada
  1. Desplegar Traefik v3 a cada node pub amb sticky sessions configurades per a Blazor Server
  2. Desplegar nginx com a proxy SSL per davant de Traefik (terminació TLS)
  3. Desplegar contenidors de prova (o l'app AutoCo si ja està migrada) darrere Traefik
  4. Configurar Keepalived a pub-01 (MASTER) i pub-02 (BACKUP) per al VIP 192.168.0.200
  5. Provar failover: systemctl stop keepalived a pub-01 → la VIP ha de moure's a pub-02 en <5 s

Configuració Keepalived (pub-01 MASTER)

vrrp_instance VI_PUB {
    state MASTER
    interface eth0
    virtual_router_id 51
    priority 110
    advert_int 1
    authentication {
        auth_type PASS
        auth_pass autoco2026
    }
    virtual_ipaddress {
        192.168.0.200/24
    }
    track_script {
        chk_nginx
    }
}
Verificació: ip addr show eth0 | grep 192.168.0.200 ha d'apareixer a pub-01. Aturant nginx a pub-01, la VIP ha de migrar a pub-02.

Harbor Registry — contenidor aïllat

Registry privat per a imatges Docker · Accés des de la LAN
Temps estimat: 1-2 h Downtime: cap Prerequisit: Fase 1 completada (independent de les altres fases)
  1. Descarregar l'instal·lador de Harbor: wget https://github.com/goharbor/harbor/releases/latest/download/harbor-offline-installer.tgz
  2. Configurar harbor.yml: hostname 192.168.0.103, certificat SSL auto-signat (o Let's Encrypt intern)
  3. Executar ./install.sh → arrenca ~8 contenidors automàticament
  4. Accedir a https://192.168.0.103 amb les credencials inicials (admin / Harbor12345 → canviar)
  5. Crear projecte privat autoco a la UI de Harbor
  6. Des de la màquina de desenvolupament: docker login 192.168.0.103 i fer push d'una imatge de prova
  7. Als nodes pub: docker pull 192.168.0.103/autoco/web:latest per verificar pull intern
Certificat SSL Harbor: per a docker login sense errors, afegir el certificat auto-signat com a CA de confiança a cada màquina que faci push/pull: /etc/docker/certs.d/192.168.0.103/ca.crt

Accés exterior — NAT + DDNS + Let's Encrypt

AutoCo.ddns.net:4443 accessible des d'internet
Temps estimat: 1 h Downtime: ~2 min (canvi SSL) Prerequisit: Fase 3 activa i domini DDNS configurat
  1. Al router: afegir regla NAT TCP 4443 → 192.168.0.200:443
  2. Al router: afegir regla NAT TCP 80 → 192.168.0.200:80 (necessari per al challenge ACME)
  3. Configurar el client DDNS per actualitzar AutoCo.ddns.net amb la IP pública del router
  4. Activar Let's Encrypt a Traefik: --certificatesresolvers.le.acme.httpchallenge.entrypoint=web
  5. Actualitzar la variable d'entorn: APP_WEB_URL=https://AutoCo.ddns.net:4443
  6. Verificar: accedir a https://AutoCo.ddns.net:4443 des de fora de la xarxa de l'escola
IP dinàmica: si l'escola no té IP pública fixa, el client DDNS ha de córrer com a servei al router o a un dels nodes Proxmox, actualitzant el registre DNS automàticament quan canviï la IP.

Migració AutoCo — branca ha-proxmox

SQL Server → PostgreSQL · DataProtection → Redis · Harbor deploy
Temps estimat: 4-6 h Downtime: ~30 min (migració de dades) Prerequisit: Totes les fases anteriors actives
  1. Crear branca: git checkout main && git checkout -b ha-proxmox
  2. web/Program.cs: canviar DataProtection de filesystem a Redis (PersistKeysToStackExchangeRedis)
  3. api/AutoCo.Api.csproj: substituir Microsoft.EntityFrameworkCore.SqlServer per Npgsql.EntityFrameworkCore.PostgreSQL
  4. api/Program.cs: canviar UseSqlServer per UseNpgsql
  5. Esborrar les migracions existents i regenerar-les per a PostgreSQL: dotnet ef migrations add InitialPostgres
  6. Construir imatges i pujar a Harbor: docker build + docker push 192.168.0.103/autoco/web:ha-1.0
  7. Migrar dades existents de SQL Server a PostgreSQL (script de migració o exportació CSV → reimportació)
  8. Actualitzar docker-compose.yml als nodes pub amb les noves imatges de Harbor
  9. Verificació final: login d'alumne, creació d'activitat, enviament d'avaluació
La branca ha-proxmox rep els canvis de main periòdicament amb git merge main. Els únics fitxers amb possible conflicte són els 3 modificats en els passos 2-4.
